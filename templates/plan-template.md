# プランタイトル

<!--
  ステータス判定ルール（assembler `resolveStatus`）:

  チェックボックスによるステータス判定は「そのセクション自身の本文」が対象です。
  ### サブフェーズのチェックボックスは親 ## フェーズには伝播しません。

  フェーズ（## H2）のステータスを確実に設定するには:
    a) 見出し行にマーカーを付ける（「✅ 完了」「進行中」など）← サブフェーズがある場合は必須
    b) サブフェーズを作らず、見出し直下にチェックリストを置く（フラット構成）

  サブフェーズ（### H3）のステータスは、そのサブフェーズ配下のチェックボックスで決まります。

  | ラベル | 判定トリガ（そのセクション自身の本文） |
  |---|---|
  | 未着手 | チェックボックスが全部 `- [ ]`、またはチェックボックスが無い |
  | 進行中 | `- [x]` と `- [ ]` が混在、または見出し/本文先頭に「進行中」「着手」 |
  | 完了   | チェックボックスが全部 `- [x]`、または見出し/本文先頭に「✅」「完了」「済」 |
  | 破棄   | Markdown からは付きません。後で report-progress スキル等で更新します。 |

  上限: フェーズ最大 50・各フェーズ本文 20,000 字・プラン全体 50,000 字。
  マーカーは「見出し行」または「本文の最初の 1 行（先頭 80 文字程度）」までしか拾われません。
-->

## フェーズ1: 設計レビュー ✅ 完了
<!--
  フェーズは見出しマーカー「✅ 完了」によって完了と判定されます。
  各サブフェーズは自分のチェックボックスが全部チェック済みなので完了と判定されます。
  サブフェーズのチェックは親フェーズには伝播しないため、見出しマーカーが正確さの要です。
-->
要件整理と設計レビューを完了させるフェーズです。

### Step 1-1: 要件ヒアリング
- [x] ステークホルダーとの面談を実施
- [x] 要件定義ドキュメントを作成

### Step 1-2: 設計レビュー
- [x] アーキテクチャ図のレビュー
- [x] レビュー指摘の反映

## フェーズ2: 実装 進行中
<!--
  フェーズは見出しマーカー「進行中」によって進行中と判定されます。
  サブフェーズは各自のチェック状態で判定されます（Step 2-1 = 進行中、Step 2-2 = 未着手）。
  サブフェーズのチェックは親フェーズには伝播しません。
  見出しマーカーを削除するとフェーズ本体のチェックボックスは 0 個のため「未着手」になります。
-->
主要機能の実装を進めるフェーズです。

### Step 2-1: バックエンド API
- [x] スキーマ定義
- [x] ハンドラ実装
- [ ] 単体テスト

### Step 2-2: フロントエンド
- [ ] 画面コンポーネント
- [ ] フォームバリデーション

## フェーズ3: リリース準備
<!--
  見出し直下にチェックボックスを置いたフラット構成のフェーズ（サブフェーズなし）。
  チェックボックスがそのままフェーズのステータスを決めます。
  全部未チェックなので未着手。混在させれば進行中・全部チェックで完了になります。
  このパターンではチェックの更新だけでフェーズのステータスが自動的に変わります。
-->
ステージング検証から本番デプロイまでを担当するフェーズです。

- [ ] E2E テスト実行
- [ ] ステージング環境での動作確認
- [ ] リリースノート作成
- [ ] 本番反映と監視
